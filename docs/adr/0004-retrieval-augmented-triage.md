# ADR 0004: Retrieval-augmented triage via TF-IDF decorator

- **Status:** Accepted
- **Date:** 2026-04-24
- **Decision makers:** Core team
- **Related:** [ADR 0002 (LiteLLM)](./0002-litellm-for-triage.md)

## Context

The LiteLLM-backed classifier from ADR 0002 works, but has two structural
limitations that our operators quickly bumped into:

1. **No memory of human corrections.** When an operator overrides the AI's
   department suggestion via `/tickets/decision`, that correction is
   persisted to the database — but it never influences the next triage
   decision. The model re-suggests the same wrong routing for the next
   ticket of the same kind. The platform is "stateless by design" at the
   worst possible level.
2. **Explanations ring hollow.** The `rationale` field generated by the
   LLM is generic prose ("gehört zu Netzwerkthemen"). Operators don't
   gain trust from it because it isn't grounded in anything they know.

Both limitations could be solved by fine-tuning — but fine-tuning is
operationally heavy (re-training on a schedule, managing model versions,
explaining why the model is now biased), and it hides the reasoning from
the operator.

## Decision

Introduce a **retrieval-augmented triage step** that runs between "ticket
enters the system" and "LLM is asked for a recommendation":

1. Find the top-3 historical, human-reviewed tickets that are textually
   most similar to the new one.
2. Hand them to the LLM as a small "here's what the operator actually
   decided in similar cases" context block, in addition to the existing
   rules-based system prompt.
3. Return the retrieved cases alongside the recommendation so the UI can
   display them as clickable references.

The implementation lands as a **decorator** around the existing
`ClassifierPort`, not as a modification of the LLM adapter:

```
HTTP route
   └── RagAssistedClassifier      (app/infrastructure/ai/rag_assisted_classifier.py)
         ├── SimilarTicketsPort   (→ TfidfSimilarTicketsAdapter)
         └── inner ClassifierPort (→ LitellmClassifier)
```

The retrieval backend is a small scikit-learn pipeline:

- Corpus: every `TicketRecordModel` where `reviewed_by IS NOT NULL`
- Vectorizer: `TfidfVectorizer(ngram_range=(1, 2), strip_accents="unicode")`
- Index: `NearestNeighbors(metric="cosine")`
- Cold start: if the corpus is smaller than `MIN_CORPUS_SIZE = 3`, the
  adapter returns an empty list and the decorator degrades to plain LLM
  classification.

The fitted index lives in `app.state.similar_tickets` as a process-wide
singleton. It is rebuilt on startup, and on demand via `POST
/admin/rebuild-rag`.

## Why not sentence-transformers / embeddings / pgvector?

Considered and rejected for v1:

- **Zero new dependencies.** scikit-learn is already pinned for the
  baseline ML classifier — TF-IDF adds nothing to `requirements.txt`.
  sentence-transformers pulls in PyTorch (~800 MB), which would break
  the Render Free tier.
- **Quality is good enough at the corpus sizes we expect.** At O(10k)
  tickets, TF-IDF + cosine retrieval answers in < 10 ms with recall that
  operators subjectively find useful.
- **The port stays stable.** When we outgrow TF-IDF (multilingual recall,
  much larger corpus), the upgrade is a single new adapter implementing
  `SimilarTicketsPort` — the decorator, the LLM adapter, the routes, and
  the UI are untouched.

## Why not a full "ticket-triage agent"?

We also considered replacing the LLM call with a tool-using agent loop
(LangGraph / custom ReAct loop). Rejected because:

- **The failure modes we actually see** (no memory, thin explanations)
  are solved by retrieval, not by multi-step reasoning.
- **Agent loops are hard to test deterministically.** Our test pyramid
  relies on fast, stubbed classifier tests — an agent loop fights that.
- **Latency and cost would 2–5×** without a clear operator benefit.

Tool-using agents remain attractive for **future** work — specifically:
clarifying-question loops when confidence is low, and workload-aware
assignment that queries team availability. Those land better as a
separate adapter once they're actually needed.

## Consequences

**Positive**
- Every corrected routing decision silently improves every future
  suggestion — no retraining step, no model-versioning complexity.
- The UI shows **three concrete past tickets** under each AI recommendation,
  which operators can click through to verify. This is a much stronger
  trust signal than a paragraph of LLM-generated rationale.
- Graceful degradation is built in. When retrieval fails (cold index, DB
  hiccup), the pipeline falls back to plain LLM classification — it
  never takes triage down.
- Zero new external dependencies. No vector database, no GPU.

**Negative**
- **Cold start** is real: during the first ~20 reviewed tickets, the
  retrieval layer contributes nothing and the preview popup shows only
  the LLM's original recommendation. We ship a `scripts/seed_demo_tickets.py`
  to paper over this for demos.
- **The index is fitted in-process.** With more than one app instance
  (horizontal scaling), each instance holds its own copy. Acceptable for
  the current scale; would need a shared store if we scale out.
- **Rebuild isn't automatic on every new review decision.** Operators
  have to hit `/admin/rebuild-rag` or restart the process to pick up the
  most recent review. Considered fire-and-forget auto-rebuild but
  rejected for v1 — it's easy enough to add when real usage asks for it.

## Implementation notes

- Domain object: `SimilarCase` (`app/domain/entities/similar_case.py`) —
  pure dataclass, no framework dependency.
- Port: `SimilarTicketsPort` — two methods, `find_similar` and `rebuild`.
- `ClassifierPort.analyze` now accepts an optional `similar_cases`
  parameter (backwards-compatible; other classifiers ignore it).
- The LLM prompt is not rewritten — the retrieved cases are injected as
  a **second** system message, which keeps the core routing rules
  unchanged and makes A/B evaluation trivial.
- `model_version` strings now gain a `+rag` suffix when retrieval was
  used, so telemetry can distinguish RAG-augmented from plain responses.

## Testing

- **Unit**: `tests/unit/test_tfidf_similar_tickets.py` — adapter tested
  against in-memory SQLite fixtures covering empty corpus, sub-threshold
  corpus, topical ranking, similarity floor, and incremental rebuild.
- **Unit**: `tests/unit/test_rag_assisted_classifier.py` — decorator
  contract: forwards cases, preserves caller-supplied cases, survives
  retrieval failures.
- **Integration**: `tests/api/test_rag_integration.py` — full HTTP flow
  with seeded corpus, stubbed LLM, real SQLite.
- **Frontend**: `SimilarCasesList.test.jsx` — component rendering,
  click-through contract.
